gRPC: Beyond Unary RPC

The RPC Abstraction

  • The core goal: Make a remote procedure call (RPC) feel exactly like calling a local function.
  • The gRPC framework handles the complex "under-the-hood" work for you.
  • Key Components:
    • Protocol Buffers (.proto): A strict, language-agnostic API contract that defines services and messages.
    • Serialization: Efficiently converts your data objects into a compact binary format for network transfer.
    • Stubs (Client) & Skeletons (Server): Auto-generated code that hides the networking logic, providing a simple function-call interface.

The Engine - HTTP/2

gRPC is built on HTTP/2, which provides the features needed for high-performance communication.

One TCP Connection

The "Highway". A single, persistent connection reduces setup latency.

Streams

The "Lanes". Independent, bidirectional channels for each RPC call.

Frames

The "Vehicles". The smallest unit of communication (HEADERS, DATA). Each frame is tagged with a Stream ID.

This structure enables true multiplexing, solving HTTP/1.1's Head-of-Line blocking.

  • Stream 3 (e.g., a slow database query) and Stream 5 (a quick health check) start at the same time.
  • Frames for both streams are interleaved on the wire. Each frame is tagged with its Stream ID.
  • Stream 5 can complete and return its result without waiting for the slow Stream 3 to finish.

One request, many responses. Ideal for subscriptions, notifications, or sending a large dataset in chunks.

Bidirectional Streaming

A true, full-duplex conversation. Ideal for chat, real-time gaming, and interactive sessions.

rpc StartChat(stream Request) returns (stream Response) {}
  • This is the native, unconstrained behavior of an HTTP/2 stream. The other RPC types are just special cases of this.
  • Client and Server can read and write messages independently, in any order.
  • Each side closes its stream independently when it's done sending messages.
  • Key concept: gRPC is message-oriented, not byte-oriented. You must chunk large data (like files) into discrete messages yourself.

Trade-offs & Limitations: Load Balancing

Long-lived connections and streams create a major challenge for traditional L4 load balancers.

  • The Problem: A standard L4 (TCP-level) load balancer sees only one long-lived connection per client.
  • All RPCs over that connection get "pinned" to a single backend server, defeating the purpose of load balancing and potentially causing overload.

Solution 1: Client-Side Balancing

The client is aware of all backend instances (via a service like Zookeeper) and chooses one per RPC. Efficient, but pushes complexity to every client.

Solution 2: L7 Proxy / Service Mesh

An intermediate proxy (like NGINX Ingress, Envoy, Linkerd) understands gRPC/HTTP2. It can inspect individual RPCs and distribute them across backends. The industry-standard approach.
